## expo-external-player — dead code on iOS, confirmed intentionally

`expo-module.config.json` declares `"platforms": ["android"]` and there is no `ios/` directory at all. `requireOptionalNativeModule` on iOS therefore resolves to `null`, so `ExpoExternalPlayerModule.ts`'s `ExpoExternalPlayer.open()` always resolves `false` on iOS if ever called. However `src/lib/player/external-players.ts:openExternalPlayerURL()` only calls `ExpoExternalPlayer.open` inside `if (Platform.OS === "android")`; the iOS path is fully handled via `Linking.canOpenURL`/`Linking.openURL`. So the module is dead code on iOS but is *correctly* gated — not "half-wired". Per exclusion (a)/(d) I'm not filing this as a bug, just recording the determination the task asked for.

## Findings kept (see StructuredOutput for full text)

1. **[crash]** `ExpoDownloadManagerModule.swift` — `taskInfoByIdentifier` / `taskIdentifiersByDownloadId` / `lastProgressTimeByIdentifier` (plain `[Int: ...]`/`[String: Int]` Dictionaries, lines 86-88) are mutated with zero synchronization from two different concurrent execution contexts: (a) the `Function`/`AsyncFunction` bodies (`startDownload`, `cancelDownload`, `cancelDownloadById`, `cancelAllDownloads`, `getActiveDownloads`) which Expo Modules Core invokes off whatever queue the JS call arrives on, and (b) `URLSessionDownloadDelegate` callbacks (`didWriteData`, `didFinishDownloadingTo`, `didCompleteWithError`) which fire on the session's own private delegate queue created by `URLSession(configuration:delegate:delegateQueue: nil)` (line 307) — a *different*, OS-managed serial queue. Swift `Dictionary` is not thread-safe for concurrent mutation. A user cancelling a download at the same moment a progress/completion callback lands (very plausible — cancel is a common user action mid-download) races `cleanup()`'s `removeValue(forKey:)` against `info(for:)`'s insert or `handleProgress`'s dictionary read, which can corrupt the dictionary's internal storage and crash with `EXC_BAD_ACCESS`/`Fatal error: Dictionary` variants of that family. No lock, serial queue, or actor isolation anywhere in the file.

2. **[bug]** `ExpoOfflineLoggerModule.swift` — an uncaught `NSException` writes a rich crash record (name/reason/full stack, `expoOfflineLoggerExceptionHandler`, lines 91-99) via `NSSetUncaughtExceptionHandler`, but the ObjC runtime always calls `abort()` after the top-level exception handler returns for an uncaught exception, which raises `SIGABRT`. This app also installs a `SIGABRT` handler (line 154) that calls `ExpoOfflineLoggerStorage.writeCrash` again with only `timestamp=...\nsignal=6` — and `writeCrash` (line 57) does `body.write(to: fileURL, atomically: true, ...)`, which *replaces* the whole file, not append. So for the single most common category of native crash in an RN/Expo app (uncaught `NSException`, e.g. index-out-of-bounds from a bridged array/dict access), the detailed report is unconditionally clobbered by the terse signal-only one a few microseconds later, and the user-facing "last native crash" (surfaced via `getOfflineCrashText()`/logs screen) never shows the exception name, reason, or stack — only "signal=6". This defeats the stated purpose of the feature for its most likely trigger.

3. **[crash]** Same file — both signal handlers (`expoOfflineLoggerSignalHandler`, lines 101-109, installed for SIGABRT/SIGILL/SIGSEGV/SIGFPE/SIGBUS/SIGPIPE) do non-async-signal-safe work inside a POSIX signal handler: `ISO8601DateFormatter` construction/formatting, Swift string interpolation, `FileManager` calls, and an atomic `Data`/`String` file write (which itself allocates, creates a temp file, and renames). None of this is on the async-signal-safe function list (basically just raw `write()`/`_exit()` and friends are safe). If the crash that raised the signal happened while a malloc/heap lock was held (a very common scenario — most SIGSEGV/SIGBUS crashes happen mid-allocation-adjacent code), re-entering the allocator from the signal handler deadlocks the process instead of producing a report, or can itself fault and mask the original crash. This can turn an app crash-with-report into an app hang with no report at all.

4. **[perf]** `ExpoDownloadManagerModule.swift` — `didCompleteWithError` (lines 61-68) forwards straight to `handleError` and never inspects `(error as NSError).userInfo[NSURLSessionDownloadTaskResumeData]`. Every download task is created fresh via `session.downloadTask(with: request)` (line 142, no `resumeData:` initializer used anywhere), and `cancelDownload`/`cancelDownloadById`/`cancelAllDownloads` call plain `task.cancel()` (never `cancel(byProducingResumeData:)`). So *any* interruption — a genuine network drop, or the app's own "pause on background" flow (`pauseActiveAnimeDownload` in `src/lib/downloads/download-manager.ts:230`, which calls `cancelManagedNativeDownload` → native `cancelDownloadById`) — discards all downloaded bytes; the JS side's only recovery path is `startManagedNativeDownload` from byte 0 (confirmed in `download-manager.ts` retry/resume paths, which never pass partial-byte state to the native `startDownload` call — there is no such parameter in the native API at all). For anime episode files (hundreds of MB to multiple GB), this means every backgrounding while `backgroundDownloading` is disabled, or any flaky-network drop, re-downloads the entire file rather than resuming, which is a meaningful mobile-data/completion-time regression relative to what "resume" implies to a user toggling a "pause"-labeled setting.

## Near-misses / considered and rejected

- **Race between `getActiveDownloads()` check and event-listener registration in `attachManagedNativeDownload`** (native-download.ts / download-manager.ts): traced the full resume path — `getManagedNativeDownload` calls native `getActiveDownloads`, and if the native task already completed/errored before the app got a chance to resubscribe, `taskInfoByIdentifier` was already cleaned up so `activeNativeDownload` is `undefined`; the JS side falls back to checking `destFile.exists` and either marks-completed or restarts. This is self-healing in practice (confirmed by reading `processQueuedAnimeDownload` end-to-end). Not filing — no concrete broken outcome found, just an internal implementation detail.
- **Early-boot event loss**: if the background URL session delivers `didFinishDownloadingTo`/`didCompleteWithError` before JS has mounted `useDownloadQueueResumeService` and called `ensureNativeDownloadListeners()`, the `sendEvent` call has no JS-side listener and the event is lost. Traced through to confirm the same self-healing fallback (`getActiveDownloads` + file-existence check) recovers a *completed* download correctly; a download that failed before JS boot loses its specific error message but is silently retried from scratch on next resume rather than hanging. Low impact given #4 above already covers the "restarts from scratch" cost; not filing as a separate finding to avoid double-counting the same root symptom.
- **Native crash handler doesn't chain a previously-installed `NSUncaughtExceptionHandler`** (`installCrashHandlers`, line 153): unlike the JS-side `installGlobalErrorHandler()` in `src/lib/offline-logger.ts` which explicitly saves and re-invokes `previousHandler`, the Swift installer calls `NSSetUncaughtExceptionHandler` unconditionally with no `NSGetUncaughtExceptionHandler()` save/chain. No other exception-handler-installing native SDK exists in this repo today (no Sentry/Bugsnag et al.), so there's no live handler being clobbered right now — capped at low/smell per instructions since I can't trace a concrete present-day failure, and left out of the top findings list in favor of the four above.
- **`destinationPath` not validated to stay under the sandbox Documents dir** in `ExpoDownloadManagerModule.swift`'s `destinationURL(from:)`/`handleDownloadComplete`: the native module will `moveItem` to whatever path string it's given with no containment check. Today `destinationPath` is always constructed by trusted app code (`download-manager.ts`'s `getEpisodeDestinationFile`, filename sanitized via regex) — no reachable attacker-controlled input today. Noted as defense-in-depth gap only, not filed (no concrete failure scenario, iOS sandbox limits blast radius to the app's own container regardless).
- **`expo-download-manager` `LSApplicationQueriesSchemes`/`UIBackgroundModes` app.config check**: verified no `UIBackgroundModes` entry is actually required for `URLSessionConfiguration.background(withIdentifier:)` transfers or the `handleEventsForBackgroundURLSession` wake callback — that's a separate OS mechanism from `UIBackgroundModes: fetch/processing`. Only `"audio"` is declared and that's unrelated/correct for the player. Ruled out.
- **`expo-offline-logger` PII/secrets audit**: `append()`/`native.log` almost never fires (see above). The one entry it does write (`capturePreviousNativeCrash`) is redacted at export time via `redactSensitiveText()` in `offline-logger.ts` before being surfaced to the user/support flow — tokens, auth headers, and home-directory usernames are stripped. Crash text itself (exception name/reason/stack, or `signal=N`) is program-generated, not user secrets. No PII/secrets-logging issue found in this module.
- **`ExpoDownloadManager` duplicate-start / cancel-then-progress ordering** (`cleanup(taskId:)` called synchronously in `cancelDownload`/`cancelDownloadById` before the async `getAllTasks`-driven `task.cancel()` actually lands): traced through `info(for:)`'s `taskDescription` fallback and confirmed it's self-healing (re-populates the maps, then `handleError`'s eventual `cleanup` removes them again) — folded into finding #1 as part of the same "no synchronization" root cause rather than filed separately.
