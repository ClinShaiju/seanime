# Tenji iOS app audit - findings (2026-07-10)

Audit of the Tenji iOS client at HEAD `65f8baf` (v0.1.24), repo `H:/Projects/seanime-tenji` (all file paths below are relative to that repo). Method: **18 disjoint sweep scopes** -> semantic dedup (63 raw findings) -> **adversarial verification** — a 3-lens verifier panel for anything the sweep rated critical/high, a single adversarial refuter otherwise. **Confirmed** = verifier(s) traced the concrete defect in the code at HEAD. Findings whose verifier was lost to API errors would carry a **PLAUSIBLE** label; none needed it this pass — every surviving finding is CONFIRMED. Raw per-scope notes live in `raw/`.

Severity counts (confirmed, post-verification): **0 critical, 3 high, 19 medium, 30 low** — 52 findings. 5 findings were refuted and dropped (appendix at bottom).

**TL;DR.** No criticals survived verification. The 3 highs are all on the native side: two memory-safety races that intermittently crash the whole app (an mpv wakeup-callback **use-after-free** on every player teardown, and **unsynchronized Swift dictionaries** in the download module mutated from two queues), plus an **unbounded-memory OOM vector** in the manga reader's default long-strip mode (every page of a chapter mounts simultaneously — the Android-only image windowing is a no-op on iOS, the only shipped target). The medium tier is dominated by silent-failure plumbing: the **WS reconnect is silently skipped** after login/logout while the app still reports "connected" (killing the whole event plane until a background/foreground cycle), **failed video loads spin forever** (iOS never handles `MPV_EVENT_END_FILE` and has no error delegate at all), and the **session bearer token ships in cleartext** through the diagnostic-log export that's explicitly built for sharing. Verification was strict: of the sweep's 15 critical/high ratings only 3 survived at high; the rest were demoted (mostly on reachability/latency grounds — demotion rationale noted inline per finding).

# HIGH

## H1 — mpv wakeup callback can fire into freed MPVLayerRenderer memory (use-after-free) on every player teardown · CONFIRMED (3-lens panel)
`modules/expo-mpv-player/ios/MPVLayerRenderer.swift:183`. `start()` registers mpv's wakeup callback with `Unmanaged.passUnretained(self)` as the context pointer. `stop()` nils `self.mpv` synchronously but only *schedules* `mpv_set_wakeup_callback(handle, nil, nil)` + `mpv_terminate_destroy(handle)` via `queue.async`, and the queued block captures only `handle` — nothing retains the renderer. The owner view's `deinit` (`MpvPlayerView.swift:103-110`) calls `stop()` then drops the last strong reference, so ARC frees the object before the queued unregister runs; mpv is still decoding and firing wakeups (property changes stream on essentially every frame/cache update), and a wakeup in that window resolves a dangling pointer and calls `processEvents()` on freed memory. Demoted from the sweep's critical: the window is typically sub-millisecond GCD enqueue-to-execute latency on a near-idle serial queue -> a probabilistic, intermittent native crash on player close, not a deterministic one, with no attacker control.

- **Fix:** call `mpv_set_wakeup_callback(handle, nil, nil)` synchronously in `stop()` before returning (before `mpv` is nil'd / dealloc can proceed); `terminate_destroy` can stay async on `queue`.

## H2 — Unsynchronized dictionary mutation across delegate queue and JS-call queue can crash the download module · CONFIRMED (3-lens panel)
`modules/expo-download-manager/ios/ExpoDownloadManagerModule.swift:86`. `taskInfoByIdentifier`, `taskIdentifiersByDownloadId`, and `lastProgressTimeByIdentifier` are plain Swift Dictionaries with no lock/queue/actor protection, mutated from two genuinely concurrent contexts: the URLSession delegate's private serial queue (`delegateQueue: nil` at `:307` — `handleProgress`/`handleDownloadComplete`/`handleError`/`info(for:)`/`cleanup`) and the Expo JS-call contexts (`startDownload` writes at `:146-147`; `cancelDownload`/`cancelDownloadById` run `cleanup` synchronously on the JS thread; `cancelAllDownloads` does `removeAll`). Concurrent Swift Dictionary mutation is UB -> EXC_BAD_ACCESS crashing the whole app. Real flows exercise the race continuously: manga chapter downloads run 4 concurrent page downloads (delegate-queue cleanups racing JS-side starts across dozens of pages), and the app-backgrounding cancel path races the active download's streaming progress callbacks. Demoted from critical: probabilistic timing-dependent crash confined to the downloads feature, no security/data-corruption dimension (background URLSession state survives relaunch).

- **Fix:** protect all reads/writes of the three dictionaries with a single serial DispatchQueue (or NSLock), or hop the delegate callback bodies onto the same queue the Function/AsyncFunction bodies use.

## H3 — Manga reader full-zoom long-strip path has no page-count cap and no image windowing on iOS · CONFIRMED (3-lens panel)
`src/components/features/manga/reader/manga-reader-screen.tsx:212`. `useFullLongStripZoom` gates purely on aspect-ratio heuristics, never chapter length — and it passes in the *common* case: LONG_STRIP is the default reading mode and the client deliberately requests page dimensions online. When true, every spread renders via a plain `.map()` (`:686-708`) inside a non-virtualized ScrollView (native pinch-zoom path, no `removeClippedSubviews`). The one per-item gate, `shouldRenderLongStripImages`, returns `true` unconditionally unless `Platform.OS === "android"` (`use-manga-reader-android-long-strip.ts:50-53`) — and iOS is the only shipped target, so there is zero image windowing on-device. Native image memory grows linearly and un-evictably with chapter length (~7-8 MB decoded per page at 3x; a 100+ page chapter ~ 700-800 MB in one ScrollView) — a real jetsam/OOM crash vector for long chapters, plus all page fetches fire simultaneously on open. Panel kept this at high.

- **Fix:** extend the Android image-windowing to iOS, or add a page-count cap that forces the virtualized FlashList path for long chapters regardless of aspect-ratio profile.

# MEDIUM

## M1 — WebSocket reconnect silently skipped on login/logout/server-url change; state keeps saying "connected" · CONFIRMED (3-lens panel)
`src/api/components/websocket-provider.tsx:123`. The connect effect's guard only reconnects when `!socket || readyState === CLOSED`, but on any dependency change (sessionToken flip at login/logout, serverUrl/authToken/protocol edit) React runs the old cleanup — whose `close()` sets the socket to CLOSING *synchronously* — then the new effect in the same commit; nothing nulls `websocketAtom`, so the guard sees a non-null CLOSING socket and skips `connectWebSocket()` deterministically. The cancelled old effect's close-listener no-ops, so `websocketConnectedAtom` never flips false either: the entire event plane (toasts, playback signaling, watch rooms, and the debrid reconnect-resume in `src/lib/player/debrid-reconnect.ts`, which arms on that atom) is silently dead while the UI reports connected, until a background->foreground cycle. Verifiers even found the finding *understates* it — the client-identity recovery path can't reconnect either. Demoted high->medium: triggers are rare on a persisted-session single-device client, and iOS backgrounding (frequent) fully self-heals.

- **Fix:** treat `readyState >= CLOSING` as reconnectable (or null/replace the socket atom and flip connected state in cleanup) so a dep change always reconnects.

## M2 — Per-user session bearer token leaks in cleartext through the offline-log export/share flow · CONFIRMED (3-lens panel)
`src/api/components/websocket-provider.tsx:71` + `src/lib/offline-logger.ts:99-108`. The WS connect routine logs the full `/events` URL including `session=<raw token>` (a 30-day, full-account bearer credential) on every mount, reconnect, and foreground; entries persist unredacted to MMKV whenever the "Enable Logging" toggle is on. The export-time `redactSensitiveText` allowlist (`token|proof|auth|authorization|access_token|refresh_token|api_key|apikey`) omits `session`, so "Copy Diagnostic Logs" scrubs the sibling `token=` param but ships `&session=<raw>` — a false sense of sanitization in the one flow explicitly built for sharing with support. Demoted high->medium: requires the opt-in toggle plus a manual export/share, and on password-protected servers the leaked token isn't standalone-usable (the first-factor password hash/HMAC token *is* redacted, and the ws/auth middlewares require it).

- **Fix:** add `session` to the redaction regexes (query + JSON-key), and preferably stop logging credentialed URLs at all.

## M3 — Server auth token and per-user session token stored in plaintext MMKV instead of the iOS Keychain · CONFIRMED
`src/atoms/storage.ts:4`. `createMMKV({ id: "seanime-mobile" })` has no `encryptionKey`, and both `serverAuthTokenAtom` and `sessionTokenAtom` (`src/atoms/server.atoms.ts:10-15,51-54`) persist through it. The server auth token is a static SHA-256 of the server password sent unchanged as `X-Seanime-Token` on every request — a permanent, non-expiring bearer equivalent to the password itself (and additionally the HMAC secret for signed stream URLs); the session token is the login bearer. `expo-secure-store` is not a dependency anywhere. Anyone with an unencrypted device backup or a jailbroken/compromised device reads both tokens straight out of the MMKV file.

- **Fix:** move both credentials to the Keychain (or give MMKV an encryption key that itself lives in the Keychain).

## M4 — Failed video loads (dead links, network errors) leave the player on an infinite spinner — no onError path exists on iOS · CONFIRMED (3-lens panel)
`modules/expo-mpv-player/ios/MPVLayerRenderer.swift:376`. `handleEvent` never handles `MPV_EVENT_END_FILE` (falls to `default: break`); `isLoading` is only cleared by FILE_LOADED / PLAYBACK_RESTART / paused-for-cache, none of which fire when `loadfile` fails outright (expired/dead debrid URL, DNS failure). The iOS delegate protocol has no error callback at all — the only `onError` emission is renderer-*creation* failure — while Android's renderer delegate does have one (parity gap). JS (`use-mpv-player.ts`) has no watchdog, so a common failure class renders as an indefinite spinner indistinguishable from buffering. Demoted high->medium: the back button stays functional during the stuck state, so recovery is back-out-and-retry; no crash.

- **Fix:** add an `MPV_EVENT_END_FILE` case, inspect `mpv_event_end_file.reason`, clear `isLoading`, and add an error delegate path so JS can transition to `status: "error"`.

## M5 — Every failed mutation (and most queries) runs two toast layers; 45s timeouts toast despite deliberate suppression · CONFIRMED (3-lens panel)
`src/api/client/requests.ts:170`. `buildSeaQuery`'s catch toasts the normalized error then rejects; `useServerMutation`'s `onError` (`:200-214` — no `muteError` field exists, no abort check) and `useServerQuery`'s error effect (`:266-278` — `muteError` opt-in only, 5 call sites) toast the same rejection again. Abort suppression is defeated: `normalizeSeaError` erases the AbortError name and `isConnectivityFailure` excludes aborts, so a 45s timeout — which the inner layer deliberately keeps silent — still produces a "Server Error: Aborted" toast. Demoted high->medium: react-native-toast-message replaces in place, so users see one (conflicting/wrong-message) toast rather than two stacked; the residual is redundant layering, unsuppressible mutation toasts, and timeout noise.

- **Fix:** toast in exactly one layer; plumb `muteError` through mutations; mark aborted rejections so outer layers can skip them.

## M6 — buildSeaQuery hard-codes JSON both ways, breaking the app's only FormData upload and only binary download · CONFIRMED
`src/api/client/requests.ts:110`. Request bodies are always `JSON.stringify`'d and responses always read via `text()` + `JSON.parse` — no FormData or binary branch. `useDecompressIssueReport` types its variable as FormData for a multipart upload the server expects via `c.FormFile("file")` (can never work as written); `useDownloadIssueReport` hits an endpoint streaming raw `application/zip` (UTF-8-mangled by `text()`). Both hooks are dead code today (zero call sites) — rated medium per the T1 ledger precedent: breaks immediately once wired to UI, and there is currently *no* supported request-layer path for any file upload or binary download.

- **Fix:** add FormData (skip stringify/Content-Type) and blob/arrayBuffer branches to `buildSeaQuery`.

## M7 — sortedCollection useMemo mutates the shared react-query cache object for the library collection · CONFIRMED (3-lens panel)
`src/hooks/use-anime-library-collection.ts:55`. When `data.stream` is populated (debrid/torrent-streaming — this deployment's primary mode), `currentList.entries = entries` writes synthetic stream-only entries directly into the cached, shared `data.lists` array that every subscriber of `useGetLibraryCollection()` receives (`collection.loaders.ts`, `unmatched.tsx`, `lib/player/session.ts`). Upstream seanime-web does the identical merge immutably and has this exact mutating line commented out — the fork kept a deliberately-abandoned pattern. Demoted high->medium: the mutation is idempotent, never reaches MMKV persistence, and no user-visible failure was traced; the real costs are permanently defeated structural sharing (re-render churn on every invalidation) plus a latent shared-state trap for future `.lists` consumers.

- **Fix:** build a new list object (`{ ...currentList, entries }`) into a new mapped `lists` array; leave the cached `data.lists` untouched.

## M8 — PrewarmBadge, EpisodeCardList, and EpisodeListItem subscribe to the full serverStatusAtom, undermining list-item memoization · CONFIRMED
`src/components/features/anime/episode-list-item.tsx:121`. Three per-item components rendered once per visible list/grid entry call `useServerStatus()` (the full Status object) for narrow needs — PrewarmBadge needs 2 debrid flags, the other two need 5 `themeSettings` spoiler fields — the same pattern ledger item T9 fixed for MediaEntryCard. `EpisodeListItem` is explicitly `React.memo`-wrapped (`:294`) yet subscribes internally, so any unrelated Status change re-renders every visible row across the 4 screens that use it; `episode-card-list.tsx` additionally puts the full Status in its `renderEpisodeCard` deps.

- **Fix:** narrowed derived atoms (mirror `useMediaCardDisplaySettings()` in `server.atoms.ts`) for the spoiler fields and `debridSettings.{enabled,preloadNextStream}`.

## M9 — ControlsOverlay re-renders on every playback tick, unmemoized, regardless of visibility · CONFIRMED
`src/components/features/player/player-controls.tsx:117`. Plain function component (no `React.memo`), mounted for the whole playback session (visibility is an opacity/pointerEvents animation only), passed the live `state` object whose `currentTime` updates ~4x/sec — forcing a full re-render of the entire controls tree (seek-bar segments, pills, buttons, time text) per tick, even while controls are hidden and non-interactive. Sustained hot-path waste on a mobile video player.

- **Fix:** wrap in `React.memo` and isolate the ticking seek-fill/time-text into a small child so the rest of the overlay doesn't re-render every tick.

## M10 — Recursive synchronous filesystem size walk runs on the JS thread ~4x/sec while any chapter is downloading · CONFIRMED (3-lens panel)
`src/lib/downloads/manga-download-manager.ts:889` + `manga-entry-downloaded-view.tsx:53`. `Directory.size` is a synchronous native getter that recursively enumerates and stats the *entire* manga-downloads tree; the Downloaded tab recomputes it in a useMemo keyed on `allChapters`, whose identity changes on every download-progress revision bump (persisted every ~250ms / 2 pages). Whenever a Downloaded tab is mounted while any chapter of any manga downloads (the exact screen users watch during downloads — and it stays mounted `display:none` after first visit), the JS thread blocks on a full walk ~4x/sec, scaling unboundedly with library size. Demoted high->medium: needs an active download + the screen mounted, and typical walks are ~10-50 ms (recurring jank, not a freeze).

- **Fix:** recompute only on chapter completion (not per progress tick), scope to the current media (see M11), and cache.

## M11 — Manga Downloaded-tab disk usage shows the whole library's size, not this manga's · CONFIRMED (3-lens panel)
`src/components/features/manga/manga-entry-downloaded-view.tsx:53`. `getMangaDownloadDiskUsage()` returns `getMangaDownloadsDir().size` — the root directory across ALL media and providers — rendered on a single manga's tab next to a chapter count that IS correctly scoped to `mediaId`. Wrong for anyone with >=2 downloaded manga (the common case). Demoted high->medium: display-only; all delete actions on the page are correctly scoped.

- **Fix:** per-media variant using `getMediaDir(mediaId).size`; keep the global function for library-wide UI only.

## M12 — useDiscoverRecentReleases recomputes wall-clock query variables on every call, thrashing the query cache · CONFIRMED
`src/components/features/discover/discover-queries.ts:96`. `airingAt_lesser/greater` are computed inline with `new Date()` (unmemoized) and the query key is `[key, JSON.stringify(variables)]`, so any render in a new wall-clock second mints a brand-new key -> cache discard, refetch, loading flash. The trigger is guaranteed, not incidental: the Discover hero carousel auto-rotates every 10s, so once activated the "Aired Recently" row refetches and collapses to a skeleton every 10 seconds, orphaning cache entries and hammering the server's AniList proxy.

- **Fix:** memoize the bounds or round them to a coarse window (e.g. current day).

## M13 — Episode 0 is a silent playback dead-end in onlinestream · CONFIRMED (3-lens panel)
`src/components/features/onlinestream/anime-entry-onlinestream-section.tsx:49`. The play-firing effect guards `if (!controller.playRequestedEpisode) return` — a falsy check that treats episode 0 as "no request" — while the controller itself correctly uses `!== null` everywhere (state is `number | null`). Requesting episode 0 (legitimate for OVAs/prologues; the server passes provider numbers through unfiltered) fetches and resolves a source, flashes the loading overlay, then never calls `playOnlineStreamEpisode` and never cancels the request: silent non-response, deterministically unplayable. Demoted high->medium: only episode-number-0 listings in one playback path.

- **Fix:** `if (controller.playRequestedEpisode === null) return` (and the same fix for `player.tsx:709`'s falsy guard).

## M14 — Failed downloads with no matching library episode have no in-place delete button (dead-end UX) · CONFIRMED (3-lens panel)
`src/components/features/anime/episode-list-item.tsx:248`. The caller-supplied `action` node (the trash button) is nested inside the `showDetailsButton` conditional instead of being a sibling, so it never renders when details are disabled. `anime-entry-downloaded-view.tsx` passes `disableDetailsButton={!matchingEpisode}` and gates all presses on `isCompleted`, so a failed download whose episode drifted out of `entry.episodes` renders as a fully inert row — under section copy that says "Remove failed downloads here" — and "Delete All" doesn't even render in a failed-only state. Demoted high->medium: requires post-download library drift (common failures keep the match and the trash renders), and the global download-queue screen offers per-row delete/retry.

- **Fix:** render `action` as a sibling of the details-button conditional so a caller-supplied delete always shows.

## M15 — No resume-data capture: any interruption or app-background pause restarts downloads from byte 0 · CONFIRMED
`modules/expo-download-manager/ios/ExpoDownloadManagerModule.swift:61`. `didCompleteWithError` never extracts `NSURLSessionDownloadTaskResumeData`; every cancel path uses plain `task.cancel()` (not `cancel(byProducingResumeData:)`); no `downloadTask(withResumeData:)` path exists. The JS "pause on app background" flow and mid-transfer failures therefore discard all downloaded bytes of multi-hundred-MB/GB episode files — a full re-download every time, contrary to what "pause" implies.

- **Fix:** capture resume data on error/cancel, persist keyed by download id, and resume via `downloadTask(withResumeData:)`.

## M16 — SIGABRT handler overwrites the detailed NSException crash record with a useless one-liner · CONFIRMED (3-lens panel)
`modules/expo-offline-logger/ios/ExpoOfflineLoggerModule.swift:101`. The uncaught-exception handler writes a rich record (name/reason/stack); the ObjC runtime then calls `abort()`, which fires the also-installed SIGABRT handler — and `writeCrash` uses `atomically: true` (whole-file replace), so the rich report is unconditionally clobbered with `signal=6` for the most common native-crash class. The user-facing "Copy Crash Report" export only ever shows the one-liner for these crashes. Demoted high->medium: diagnostics-only impact — but this app has no other crash channel.

- **Fix:** handled-this-launch flag, append instead of atomic replace, or write from only one of the two paths.

## M17 — Signal handlers do non-async-signal-safe Foundation I/O, risking a hang instead of a crash report · CONFIRMED (3-lens panel)
`modules/expo-offline-logger/ios/ExpoOfflineLoggerModule.swift:101`. The handler installed for SIGABRT/SIGILL/SIGSEGV/SIGFPE/SIGBUS/SIGPIPE constructs an `ISO8601DateFormatter`, interpolates Swift strings, and does FileManager-backed atomic writes — all allocating, none async-signal-safe. A fault taken while the malloc lock is held (typical for heap-corruption crashes) deadlocks the handler re-entering the allocator: the app freezes instead of crashing, losing both the offline log and the Apple crash report. Demoted high->medium: only manifests during an already-fatal crash, probabilistically, in a diagnostics-only module.

- **Fix:** preformatted buffer + raw `write(2)` to a pre-opened fd in the handler; defer formatting to next launch (or drop signal hooks for MetricKit).

## M18 — Disabling Adult Content leaves previously-picked adult tags stuck active and invisible in the search filter sheet · CONFIRMED
`src/components/features/discover/search-filter-sheet.tsx:215`. The Tags MultiToggle filters its *options* to non-adult when `draft.isAdult` is off, but nothing prunes `draft.tags`: a previously-selected adult-only tag's chip vanishes with no way to deselect it, still ships in the query (contradictory `isAdult: false` + adult tag -> empty/near-empty results with no explanation), and keeps inflating the Filter badge via `getActiveFiltersCount`. Only a full Reset (or re-enabling adult content) recovers — neither is obvious.

- **Fix:** prune adult tags from draft/applied params when the adult toggle turns off.

## M19 — "Upcoming" status filter silently no-ops under "Release date" sort while the UI shows it active · CONFIRMED
`src/lib/search/search-atoms.ts:56` (duplicated for manga at `:82`). START_DATE_DESC strips NOT_YET_RELEASED from the status list; if "Upcoming" was the only status selected the filtered list is empty so `status` becomes `undefined` — no status filter at all, results span every status. The sheet re-syncs its draft from the unmodified applied params (chip stays visually selected) and the badge still counts the filter. Upstream seanime-web applies the same strip but falls back to `["FINISHED","RELEASING"]`; the Tenji port dropped the fallback.

- **Fix:** restore the upstream fallback (or disable/annotate the combination in the sheet).
