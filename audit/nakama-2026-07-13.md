## 0. TL;DR — the two bug classes behind "various bugs"

Almost every user-visible nakama symptom collapses to **two structural root causes**, not N
independent bugs:

1. **"Follower never gets a live player, so sync applies to nothing."** Caused by a
   *silent no-op path*: sync is applied only if a `<video>`/player already exists
   (`if (!videoElement) return`), and several code paths lead to "should be playing" with no
   player and no recovery — the per-room opt-out sticker (F18), the missing stream identity in
   the room payload for `file`/`onlinestream` (F8), and the fixed-id subscriber teardown race
   on the legacy path (F11).
2. **Echo / oscillation / frozen room.** Caused by the *position-report + heuristic
   echo-rejection* model: the server integrates a position feed and then has to guess which
   incoming message is a real action vs an applied echo (`echoDebounce`/`crossEcho`/
   `flipChatter`). That guessing drops genuine second-controller actions (F9) and, combined
   with sticky-but-offline controller promotion (F6), freezes the room's authoritative clock.

The independent design (§3) makes **both classes impossible by construction** (anchor/logical-
clock authority + one desired-state reducer). The comparison (§4) maps every finding to the
structural change that removes it.

Underneath those, a cluster of **genuine Go data races** (F1, F2, F3, F12, F13) exists because
`m.settings` and several connection fields are touched from many goroutines without a shared
lock. F12 (`participants` unlocked iteration) can **fatally crash the whole server process**.

## 3. Independent ideal design (one agent, from scratch)

> Produced independently with only the requirements docs + a skim of the current message surface
> as constraints — deliberately *not* a description of the current code. Reproduced verbatim.

The current system works but is held together by heuristics. Two bug classes — "follower has no
live player" and echo/oscillation — are **structural**, and the ideal design makes both
*impossible by construction* rather than debounced away. Two ideas do almost all the work:

- **A. The room is a shared logical clock ("anchor"), not a stream of positions.** Authority is
  `{seq, mediaIdentity, paused, anchorPos, anchorWall, rate}`. The server never integrates
  reported positions; it stores the last *intent* and re-derives `pos(now)`. **Everyone,
  including the controller, is a follower of the anchor.** The controller is distinguished only
  by "is allowed to write the anchor." This deletes the entire echo-rejection apparatus.
- **B. One reducer owns "what should be playing," and one lifecycle effect makes the player
  converge to it.** "Open the player" and "resolve the stream" become a single client-
  orchestrated, idempotent handshake with one owner. A follower can never be in "should be
  playing" with no live player, silently.

### 3.1 Roles & topology

The current host-is-a-whole-instance model is wrong *as the primary abstraction*: it conflates a
**hub** (backend owning pool + room registry + relay) with a **room host** (the user who created
a room), and spawns two parallel engines (`watch_party*.go` instance↔instance, `watch_room.go`
same-instance) that duplicate sync logic.

Ideal: **the room is the only unit; collapse to one engine.**
- **Hub** — exactly one backend owns the canonical pool + room registry (the VPS). "Host mode" is
  demoted from "the watch-together role" to a **transport bootstrap**: it only means "this backend
  accepts external instances' users into its pool."
- **Room** = `{id, hostUserId, controllerUserId, mediaIdentity, anchor, participants[], perms,
  settings}`. The legacy "watch party" becomes "a room with one external participant" — not a
  second engine. Delete `watch_party*.go`'s sync/relay; keep only its transport bits.
- **Identity is per-USER, not per-client.** `Participant{ userKey, sessions: map[clientId]Session
  }`. Control/promotion per-user; sync fans out to all of a user's live sessions (Denshi + Tenji
  simultaneously without one overwriting the other's slot). `userKey` = profile username,
  external users namespaced `<serverTag>-username`.

Verdict on host-is-an-instance: keep instance host/peer **only** to attach an external
instance's users to the hub pool. It must not be the sync authority.

### 3.2 Transport, reconnection, identity

**One envelope over an abstract `Link`** with three impls: `LocalClientLink` (browser/Denshi/
Tenji already on the hub's `WSEventManager`), `PeerLink` (direct ws), `RelayLink` (hosted rooms
relay for NAT). The room engine calls `link.Send(envelope)`; the link hides direct-vs-relay (the
current `ConnectionMode` split leaks into `SendMessage` — one code path instead). Identity =
persisted client secret + stable `clientId` (already shipped); external instances carry
`peerId`+`serverTag`; auth = password→HMAC (machinery already in `share.go`). **Reconnection is
trivial under the anchor model**: a client re-reads the current anchor and re-derives position —
no catch-up choreography.

### 3.3 Sync engine (the core)

**Authority = an anchor, not a position feed:**
```
Anchor { seq uint64; media MediaIdentity; paused bool; anchorPos float64; anchorWall int64; rate float64; stopped bool }
derivedPos(now) = paused ? anchorPos : anchorPos + rate*(now - anchorWall)
```
The server stores exactly one `Anchor` per room, never integrates reported positions, never
echoes computed state as authority. Clients derive their own target from `(anchor, localClock,
clock-offset)`.

**Intents, not reports.** The controller emits discrete transitions only — `Pause/Play/Seek/
SetRate/ChangeEpisode/Stop` — each bumping `seq` and producing a new anchor (the only thing
broadcast). This dissolves the echo class:
- **Followers never emit** → nothing to reject → `echoPosTol`/`echoDebounce`/`crossEcho`/
  `flipChatter`/`isBarePause`/`minPauseFlipInterval`/exclude-sender/exclude-controller **all
  deleted.**
- **The controller is a follower of its own anchor** → a controller whose player stalls/buffers/
  fires a programmatic play/pause behaves like any follower and can't poison the room.

**The one hard predicate** (because Tenji MPV can be driven outside the app UI): a player event is
a *candidate intent* iff after it the player **diverges from the applied anchor beyond tolerance**
AND is **not buffering/seeking** (`readyState ≥ HAVE_FUTURE_DATA && !seeking`). Matches anchor →
echo → drop. On the controller → emit `intent{seq: applied+1}`. On a follower → don't fight the
room, surface a passive "out of sync — resync?" affordance. This is the *only* buffering rule that
survives, and it's a genuine player state, not a timer.

**Tick vs event:** event-driven anchor broadcast on each intent (snappy) + a low-frequency
(~3-5s) keepalive re-broadcast. **No 500ms server position feed** (clients derive locally) →
removes `broadcastTickMs` and the exclude-driver special case. Follower convergence stays local
and keeps the good parts: deadband → `playbackRate` glide nudge → single hard-seek + cooldown
(the directstream seek-reset reason for `SEEK_COOLDOWN_MS` is real). A stalled controller can't
drag anyone back because the anchor keeps advancing on wall-clock.

**Clock discipline:** formalize the existing half-RTT machinery (`ws-latency.ts`) into a periodic
ping/pong `serverClockOffset` (median of N), applied in one place — replacing today's scattered
"lead/lag by half-RTT" corrections.

### 3.4 Stream sharing (cross-account, cross-player)

Keep the shared-selection requirement; harden into a **resolution ladder**. `MediaIdentity.
selection` = `{provider, torrentItemId, fileId, infohash}` (portable identity, not a raw URL):
1. **Fast path** — reuse the host's resolved URL when account-portable (TorBox CDN links are;
   today's join-stream behavior, kept as tier 1).
2. **Per-account unrestrict** — IP/account-bound links (Real-Debrid) → peer resolves
   `{infohash, fileId}` on its *own* account.
3. **Degraded** — no account / dead link → that participant enters an explicit `degraded` room-UI
   state; the room runs for everyone else. Local-file rooms are degraded-by-default for members
   without the file.

**Player-agnostic protocol:** the room protocol never mentions player type; each client maps the
resolved handle to native `<video>` (Denshi) or MPV (Tenji). `useDenshiPlayer`/`playbackType` are
client-local, never room fields (today the server tracks `useDenshiPlayer` — move it fully
client-side).

### 3.5 Player lifecycle — one reducer, one owner, atomic open (fixes follower-no-player)

One pure `desired = reduce(roomAnchor, localOptOut, capabilities)` → `stop | play(media,atPos,
paused)`. One lifecycle effect reconciles the *live player* to `desired`: `play(X)` & no live
player → run the atomic open+resolve handshake; `play(X)` & live player → apply anchor; `stop` →
tear down. **Sync applies only after the player reports ready, and there is no silent no-op path**
— the effect *guarantees* a live player converges or lands in explicit `error`/`degraded` (today
`if(!videoElement)return` is that silent no-op).

**Atomic client-orchestrated open**, always in this order: (1) client mounts the player shell
first (synchronous, client-local — the surface exists before any resolve); (2) client asks the
server to resolve → server returns **data** (URL+metadata), it does not push into a maybe-absent
player; (3) player attaches & loads. The resolve result is **cached/replayed** if the player
signals ready slightly late — **never dropped with `return nil`** (this kills the
`!IsOpenActive → return nil` race at `stream.go:481/579/702`). Idempotency keys on
`(clientId, media.seq)`. One entry (`reconcile(desired)`) serves auto-follow, manual "Join
stream", and normal playback alike, so a follower joins through the exact path a normal viewer
does.

### 3.6 Concurrency — actor-per-room

No `room.mu`. Each room is **one goroutine owning its state exclusively**; all mutation arrives as
messages on the room's inbox (`Intent/Join/Leave/ClientDisconnect/SetPerm/Keepalive/Close`). It
mutates freely (single owner → no lock), then emits an **immutable value snapshot** to the
transport — so `snapshotLocked` and the concurrent-map-crash class (F12) **cannot occur**. The
only shared lock is the registry (`map[roomId]→inbox`, small RWMutex/sharded). Each room owns its
own keepalive timer (a message to itself) and self-reaps on idle. Backpressure: per-session
bounded outbound buffer drops stale anchors (last-value-wins — only the newest matters) instead of
blocking the room, so a slow socket (F7) can't stall it.

| Invariant | Protected by |
|---|---|
| Room state never concurrently mutated | single owning goroutine (no lock) |
| Broadcast never races membership mutation | broadcast is a value copy made inside the goroutine |
| Registry lookups vs create/close | one small RWMutex / sharded map |
| Intent ordering / last-write-wins | serialized inbox + `seq` compare |
| Slow WS write can't stall the room | transport `Send` is async/per-session buffered |

### 3.7 Protocol — one source of truth, versioned

One Go file defines every wire type (envelope + intents + anchor + room-state + card), marked for
codegen → **codegen the TS types** (extend the existing `api/generated` pattern to WS payloads) so
`Nakama_RoomPlaybackStatusPayload` can't drift from Go. One discriminated-union envelope
`{v, type, payload}` shared by all three transports via a **table-driven registry** (replaces the
per-`if event.Type==` marshal blocks in `nakama.go:260-343`). `v` negotiated at connect; major
mismatch → explicit "update your client" (a hub upgrade must not silently mis-parse an old
Denshi/Tenji). `seq` = runtime ordering; `v` = schema version (kept distinct).

### 3.8 Settings — a sum type, not contradictory booleans

`Mode = Off | Hub{password} | Member{remoteURL,password}` — being both hub and member is
**unrepresentable**. Derived facts (`amHosting`, `connectedToHub`, `connectionMode`) are computed,
never stored (kills `IsHost`-vs-`hostConnection` skew and the `shouldStart/Stop/Connect/Disconnect`
thicket in `SetSettings`). Per-room settings (password, `forceHostTracks`, auto-skip vote,
lockstep) live on the **room**, not global settings. `Enabled` + per-user opt-out stay.

### 3.9 Failure modes

Controller disconnect → room keeps advancing (no freeze), auto-promote next by `joinedAt`,
original host reclaims on reconnect. Follower can't resolve → explicit `degraded`, room continues.
Partition → idempotent rejoin + re-derive. Controller stall → anchor unaffected (intents not
positions; buffering fails the readyState gate). Two controllers act → serialized by inbox, higher
`seq` wins. Version mismatch → rejected at connect, never silent mis-parse.

## Appendix B — follow-up finding: sync does not re-converge when a follower gets ahead (2026-07-13)

Triggered by a live report: host had OP/ED auto-skip on, follower didn't; the follower seeked
into the OP, the host skipped it, and they **played disjointly and never re-synced**. Two defects,
one primary.

- **F25 (PRIMARY, HIGH) — `nakama-room-sync.ts:441-468`: reconciliation is one-directional.**
  The heartbeat path hard-seeks a follower **forward** to a driver that's ahead
  (`drift > HARD_SEEK_DRIFT`, :448), but **refuses to rewind** to a driver that's behind
  (`drift < -HARD_SEEK_DRIFT`, :455 — "never rewind to a stalled driver"). That branch is a
  patch for the iOS frozen-driver rubber-band, but it conflates a *stalled* driver (frozen
  position) with one *legitimately at an earlier position* (a follower skipped/seeked ahead,
  buffered past the target, missed an emit, or the driver rewound via heartbeat). So **any
  divergence that leaves a follower ahead of the driver is permanent** — the sole general-purpose
  convergence-when-ahead is the ±5% `playbackRate` nudge, which only covers drift ≤
  `HARD_SEEK_DRIFT` (0.6 s); anything larger (an OP skip is ~80 s) lands in the dead branch. This
  is the real "underlying issue": the engine assumes only the driver's position can be authoritative
  *forward*, so it can't recover from a follower running ahead for any reason. Fix: distinguish
  stalled from behind by whether the driver's reported `currentTime` **advances** across heartbeats
  (delta ≈ wall-clock while `!paused`) → rewind to an advancing-behind driver (reuse
  `SEEK_COOLDOWN_MS`); hold only for a frozen one. Discrete seeks already snap both ways, so the fix
  is localized to the heartbeat branch. Dissolves under the §3 anchor model (the anchor advances on
  wall-clock, so a stall never moves it → nothing to rubber-band against, and everyone converges
  including rewinds).

- **F26 (side, MED) — `mpv-core-player-inner.tsx:1054`: MpvCore ignores room auto-skip
  coordination.** VideoCore (`video-core-time-range.tsx:147-154`) suppresses auto-skip + skip
  buttons for non-controller room members and uses the room vote (`effectiveAutoSkip`) for the
  controller — so only the controller skips and it propagates as a synced seek. MpvCore's auto-skip
  checks none of that (bare local `autoSkip` atom), so an MpvCore **follower** skips independently
  → diverges (the F25 trigger). Fix: apply the same `amController`/`effectiveAutoSkip` gate; more
  generally, while in a room, room-coordinated player behaviors should temporarily **override** the
  client's local settings and restore on leave. The Tenji iOS MPV client likely shares this gap
  (separate repo) — flag for parity. Secondary to F25: even with no coordination, F25 makes the room
  re-converge.

Both are tracked as tasks #26 (F25) and #27 (F26).
